### 1.1. User Story Description

> US11 - As Administrator, I want to register a vaccine.

### 1.2. Customer Specifications and Clarifications

As especificações do cliente indicam que a vacina deve ser definida pelos seus metadados comerciais e de administração, com uma restrição específica sobre a tecnologia utilizada.

* **Registo Duplo:** O processo de "registar uma vacina" exige o registo de dois conjuntos de dados críticos, que serão modelados como duas entidades de domínio:
    1.  **A Entidade `Vaccine`:** A definição comercial (Nome, Marca).
    2.  **A Entidade `AdministrationProtocol`:** As regras de dosagem e elegibilidade (Nº de doses e idade mínima/máxima).
* **Validação Estrita:** O sistema deve implementar uma validação estrita para o campo Tecnologia (AC11-1) e garantir que o Nome Comercial é único (AC11-2).

### 1.3. Acceptance Criteria

* **AC11-1 (Oficial):** A tecnologia da vacina deve ser selecionada a partir de uma lista predefinida de tipos disponíveis (cf. https://www.pfizer.com/news/articles/understanding six types of vaccine technologies).
* **AC11-2 (Equipa):** O Nome Comercial (`commercialName`) da vacina deve ser único em todo o sistema.
* **AC11-3 (Equipa):** Todos os campos de dados do `AdministrationProtocol` (Nº de doses, intervalo em dias, idade mínima, idade máxima) são obrigatórios e devem ser logicamente válidos (e.g., Nº de doses ≥ 1; Idade Mínima ≤ Idade Máxima).
* **AC11-4 (Equipa):** O Administrador deve selecionar um `VaccineType` existente para o qual a vacina está a ser registada.

### 1.4. Found out Dependencies

* **US10 - Specify a new vaccine type:** A US11 depende de um ou mais `VaccineType` já terem sido especificados e registados no sistema.

* **Entidades de Domínio:** Dependência da existência das entidades `VaccineType` e `AdministrationProtocol` para modelar corretamente a informação.
